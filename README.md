# School AI Library

A web app for secondary schools: teachers upload course material (PDF, Word,
PowerPoint), AI (via Groq) automatically generates a summary, key topics, and
key terms, and students can read it anytime from a clean, searchable library.

## Features

- Separate teacher and student accounts (teacher sign-up is gated by a code
  so random visitors can't register as staff)
- Teachers create courses (by class level, e.g. SS2) and upload outlines,
  notes, or slides
- Automatic text extraction from PDF / DOCX / PPTX
- AI summary (overview, key topics, key terms) generated by Groq's API
- Ask-a-question box on every document — students can ask follow-up
  questions and get answers grounded only in that document's content
- Students browse all courses, filter by class level, and read summaries —
  with the full extracted text available too
- Re-summarize or delete documents; delete whole courses

## Tech stack

Flask, Flask-SQLAlchemy, Flask-Login, SQLite (local) / PostgreSQL (Render),
Groq API for summarization, pypdf / python-docx / python-pptx for text
extraction.

## Project structure

```
school-ai-portal/
  app.py                  # app factory + shared routes (home, document view/download)
  config.py                # configuration from environment variables
  extensions.py             # db, login_manager instances
  models.py                 # User, Course, Document
  blueprints/
    auth.py                 # register, login, logout
    teacher.py               # course + document management, upload, AI trigger
    student.py                # browse courses, read-only views
  utils/
    file_parser.py            # extracts text from pdf/docx/pptx
    ai_summarizer.py           # calls Groq, parses structured summary
    decorators.py               # @teacher_required / @student_required
  templates/                    # Jinja2 templates
  static/css/style.css            # design system
```

## Local setup

1. **Create a virtual environment and install dependencies**

   ```bash
   cd school-ai-portal
   python3 -m venv venv
   source venv/bin/activate        # on Windows: venv\Scripts\activate
   pip install -r requirements.txt
   ```

2. **Configure environment variables**

   Copy `.env.example` to `.env` and fill it in:

   ```bash
   cp .env.example .env
   ```

   - `SECRET_KEY` — generate with `python -c "import secrets; print(secrets.token_hex(32))"`
   - `GROQ_API_KEY` — get a free key at https://console.groq.com/keys
   - `TEACHER_SIGNUP_CODE` — any code you choose; share it only with staff
   - `DATABASE_URL` — leave blank for local development (SQLite is used automatically)

3. **Run it**

   ```bash
   python app.py
   ```

   Visit http://127.0.0.1:5000

The database (`instance/school.db`) and uploaded files (`uploads/`) are
created automatically on first run.

## Deploying to Render (same setup as Merit360 / SafeSpace)

1. Push this folder to a GitHub repo (e.g. under Cuprous-Hub).
2. On Render, create a **Web Service** from that repo.
3. Build command: `pip install -r requirements.txt`
4. Start command: `gunicorn app:app`
5. Add a free **PostgreSQL** instance on Render and copy its connection
   string into the `DATABASE_URL` environment variable on the web service
   (the app automatically converts `postgres://` to `postgresql://`).
6. Add the same environment variables as your `.env` file
   (`SECRET_KEY`, `GROQ_API_KEY`, `GROQ_MODEL`, `TEACHER_SIGNUP_CODE`,
   `MAX_UPLOAD_MB`) in Render's dashboard.
7. **Important:** Render's filesystem is ephemeral on the free tier, so
   uploaded files will be wiped on redeploy/restart. For a real production
   deployment, swap the local `uploads/` folder for a persistent disk (Render
   offers paid persistent disks) or object storage like Cloudflare R2 / AWS
   S3. This is fine for demos and testing as-is.

## Notes & possible improvements

- **Long documents:** text sent to the AI is truncated to ~18,000 characters
  to stay well within context limits. For very long textbooks, you could
  chunk the text and summarize each chunk, then combine.
- **No class enrollment yet:** any logged-in student can see any course.
  If you want students restricted to their own class, add an `Enrollment`
  model linking students to courses (or filter by the student's own level —
  you'd add a `level` field to `User` for students).
- **Background processing:** summarization currently happens synchronously
  during upload (usually a few seconds). For very large files or heavy
  traffic, move this to a background task queue (e.g. Celery + Redis).
- **Search:** since `full_text` is stored per document, adding a search box
  across course material would be a natural next feature.
- **Q&A history:** answers from the "Ask a question" box aren't saved
  anywhere — they reset on page refresh. If you want a persistent study
  log per student, add a `QuestionLog` model (student_id, document_id,
  question, answer, created_at) and write to it in the `/ask` route.
